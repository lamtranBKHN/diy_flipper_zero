# NFC Scanner Performance & Robustness Plan

## Problem Summary

From debug logs, NFC read is slow in two scenarios:

1. **No card present**: ~3.5s scanning (5 rounds × 4 protocols with 150/300ms inter-round delays)
2. **Card present (NTAG203 example)**: ~8s total from "read" press to completion
   - ~580ms base protocol scanning (A/B/FeliCa/SRIX)
   - ~500ms child protocol detection (8 redundant InListPassiveTarget polls)
   - ~1.5s actual page reading (11 chunks × ~55ms per InDataExchange)
   - ~1.2s protocol support plugin loading + data parsing

Root causes:
- Every protocol detect re-sends InListPassiveTarget (Type A ~55ms each)
  → 13 polls total with card present (1 base + 3 children + 5 grandchildren + 4 others)
- Type B and FeliCa polls take ~200ms each with no card (PN532 internal RF timeout)
- Children of ISO14443-4A are probed even when SAK=0 (no ISO-DEP capability)
- `pn532_wait_ready_ms` polls I2C every 50ms, adding latency

## Changes

### 1. Cache Type A target across detects (HIGH IMPACT)

**File**: `targets/f7/furi_hal/furi_hal_nfc_pn532.c` — `furi_hal_nfc_pn532_trx_short_frame()`

Problem: `trx_short_frame` always calls `furi_hal_pn532_poll_iso14443a()` (InListPassiveTarget) even when a target is already cached and fresh from a prior poll. This causes every child/grandchild protocol detect to re-scan the same card.

Fix: If `furi_hal_nfc_pn532.target_valid == true` and target is fresh (age < `PN532_TARGET_FRESHNESS_TIMEOUT_MS` = 5000ms), skip InListPassiveTarget. Use cached ATQA, synthesize RxStart/RxEnd events, and proceed to SDD/SELECT (which already use cached data via `exchange_internal`).

Saves ~55ms per redundant poll × ~8 = ~440ms during card detection.

### 2. Skip ISO14443-4A grandchildren when SAK lacks ISO-DEP bit (HIGH IMPACT)

**File**: `lib/nfc/nfc_scanner.c` — `nfc_scanner_state_handler_find_children_protocols()`

Problem: Non-ISO-DEP cards (SAK bit 5 = 0) still probe all 5 grandchildren of ISO14443-4A (Desfire, Plus, NTAG4xx, Type4Tag, EMV), each of which does RATS → fails → detected=0. This wastes 5 × ~55ms = ~275ms.

Fix: After detecting ISO14443-3A, read the cached SAK from `furi_hal_nfc_pn532` via an accessor function. If SAK bit 5 (0x20) is not set, skip children of ISO14443-4A during children enumeration.

Requires adding a `furi_hal_nfc_pn532_get_sak()` accessor to expose the cached SAK.

Saves ~275ms for non-ISO-DEP cards.

### 3. Skip Type B/FeliCa on empty rounds (MEDIUM IMPACT)

**File**: `lib/nfc/nfc_scanner.c` — `nfc_scanner_state_handler_try_base_pollers()`

Problem: Type B and FeliCa polls take ~200ms each with no card (PN532 internal RF timeout). In the "no card" case, they never find anything.

Fix: Reorder base protocol polling to always try Type A (fast ~55ms) first. If Type A returns "no target found" in the current round, skip Type B and FeliCa for that round. Only probe them if Type A returned a valid target (extremely rare multi-protocol card).

Saves ~400ms per empty round × 5 rounds = ~2s total for the "no card" case.

### 4. Reduce empty scan threshold (MEDIUM IMPACT)

**File**: `lib/nfc/nfc_scanner.c`

Change `NFC_SCANNER_MAX_EMPTY_ROUNDS` from 5 to 3.

After 3 consecutive empty rounds (~2s total with optimizations 1+3), it's overwhelmingly likely no card is present. The UI can trigger a new scan on user action if needed.

Saves ~1s.

### 5. Fast RATS failure with no cached ATS (LOW IMPACT)

**File**: `targets/f7/furi_hal/furi_hal_nfc_pn532.c` — `furi_hal_nfc_pn532_exchange_internal()`

Problem: RATS command (0xE0) is sent via InDataExchange to PN532 even when `cached_ats_len == 0` (card is not ISO-DEP). The PN532 returns an error after ~55ms.

Fix: At the top of the RATS interception block, add a check: if `cached_ats_len == 0`, return `FuriHalNfcErrorCommunication` immediately without sending to PN532. This avoids sending a pointless InDataExchange command and waiting for the response.

Saves ~50ms per grandchild detect that reaches RATS phase (after Optimization 2 already skips most).

### 6. Reduce `pn532_wait_ready_ms` poll interval (LOW IMPACT)

**File**: `targets/f7/furi_hal/furi_hal_pn532.c` — `pn532_wait_ready_ms()`

Problem: The wait loop polls PN532 every 50ms. Since each poll is a 1-byte I2C read (~100µs at 100kHz), the interval can be safely reduced.

Fix: Change `furi_delay_ms(50)` to `furi_delay_ms(20)`. This reduces the average wait time by ~15ms per poll cycle without increasing I2C bus load significantly.

Saves ~15ms per wait × ~20 I2C transactions in a typical scan = ~300ms cumulative.

### Implementation Order

1. Optimization 1 (target caching) — biggest gain, foundational for others
2. Optimization 2 (SAK check) + Optimization 5 (fast RATS fail) — depend on 1 conceptually
3. Optimization 3 (skip B/FeliCa) — independent, purely scanner logic
4. Optimization 4 (reduce rounds) — simple constant change
5. Optimization 6 (faster polling) — independent, purely PN532 driver

### Expected Improvement (NTAG203 example)

| Phase | Before | After |
|---|---|---|
| Empty scan (no card) | ~3.5s | ~0.8s |
| Card detection | ~580ms | ~55ms (A only, 1 poll) |
| Child protocol detect | ~500ms | ~165ms (3 children only) |
| Page reading | ~1.5s | ~1.5s (unchanged) |
| **Total with card** | ~8s | ~4-5s |

### Risk Assessment

- **Optimization 1**: Low risk. If `trx_short_frame` returns cached data when target is stale, the SDD/SELECT will still detect the mismatch (UID check fails) and trigger a re-poll via the normal error path. The `target_valid` flag is cleared on mode changes and communication errors.
- **Optimization 2**: Low risk. Only affects grandchildren of 4A, which already fail. SAK bit 5 is the standard ISO/IEC 14443-4 indicator.
- **Optimization 3**: Low risk. Type B/FeliCa-only detection fails open (takes one more round to detect). Multi-protocol A+B cards are extremely rare.
- **Optimization 5**: Low risk. Early return from RATS interception when ATS not cached.
