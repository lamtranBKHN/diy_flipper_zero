# NFC Issues: MIFARE Detection Failure + Crash on Exit

## Analysis Summary

### Issue 1: MIFARE 1K/4k Not Detected
**Symptom**: Card UID detected (ATQA=0004 SAK=08 UID4) but type unknown. NTAG203 works fine.

**Root Cause** (confirmed through code analysis):

1. **Detection method**: `mf_classic_poller_detect()` at `mf_classic_poller.c:2245` sends AUTH command and expects 4-byte nonce + WrongCRC response. This is correct for MIFARE Classic.

2. **The real issue**: The "no targets found" logs appear BETWEEN successful base detection and child protocol detection cycles. The PN532 successfully detects the card during base polling but then intermittently fails to detect it during MIFARE Classic protocol probing.

3. **Key insight from `furi_hal_pn532.c:41-45`**:
   ```
   /* Note: furi_hal_i2c_bus_reset is NOT called on PN532 errors.
    * The 9 SCL pulse bus reset desyncs the PN532's I2C slave state machine,
    * causing permanent address NACK. With no hardware RST pin on this DIY board,
    * there is no way to recover once desynced. */
   ```
   This DIY board has no hardware reset pin for PN532. I2C desync can cause silent communication failures.

4. **The polling works but data exchange fails**: The logs show:
   - Card found at base layer (`poll_iso14443a: Tg=1 ATQA=0004 SAK=08 UIDlen=4`)
   - Child protocol detection starts
   - Eventually `in_data_exchange status error: 0x13` appears

5. **NTAG203 success explanation**: MF Ultralight/NTAG203 detection uses simple page reads (no authentication). MIFARE Classic requires AUTH command which involves a challenge-response that may be more sensitive to I2C timing issues.

**Affected files**:
- `lib/nfc/protocols/mf_classic/mf_classic_poller.c:2245-2272` - detect function
- `lib/nfc/nfc_scanner.c` - protocol detection flow
- `targets/f7/furi_hal/furi_hal_pn532.c` - I2C HAL (known limitations)

### Issue 2: Crash on Exit
**Symptom**: Device crashes when pressing Back to exit NFC tool (log line 76421).

**Root Cause** (probable):
Looking at `nfc_app_free()` in `nfc_app.c:142-226`:
- `nfc_free(instance->nfc)` at line 150
- Later: `nfc_protocol_support_free()` at line 158-160
- Views removed and freed
- Finally: `scene_manager_free()` and `view_dispatcher_free()`

The poller is stopped/freed in `nfc_protocol_support_scene_read_on_event` when handling back event or poller events. However, if the poller callback is still pending when cleanup happens, it could cause issues.

Also, `nfc_protocol_support_scene_read_on_exit()` (line 370-374) only resets popup and stops blink - it does NOT stop the poller. The poller is supposed to be stopped before this by the event handler, but if an event is in-flight, there could be a race.

**Affected files**:
- `applications/main/nfc/nfc_app.c` - cleanup order
- `applications/main/nfc/helpers/protocol_support/nfc_protocol_support.c` - scene exit handlers

## Investigation Findings

### Finding 1: I2C Communication Fragility (Confirmed)
The DIY board has known I2C issues with PN532:
- No hardware reset pin for PN532
- Comment in `furi_hal_pn532.c:41-45` warns that I2C bus reset would cause permanent desync
- Single retry only (`PN532_I2C_RETRIES = 1`)
- 2ms delay before sending frames (`furi_delay_us(2000)`)

The "no targets found" during MIFARE detection but success for base protocol suggests the card responds to SENS_REQ but fails during the more timing-sensitive AUTH exchange.

### Finding 2: PN532 Error 0x13
The `in_data_exchange status error: 0x13` in logs comes from `furi_hal_pn532.c:581`:
```c
FURI_LOG_W(TAG, "in_data_exchange status error: 0x%02X", response[1]);
```
Error 0x13 is not explicitly handled (0x01=timeout, 0x14=auth error), so it returns `FuriHalPn532ErrorComm`.

### Finding 3: MIFARE Classic Detection Logic (Correct)
The `mf_classic_poller_detect` function at line 2245 is correct:
- Sends AUTH command expecting 4-byte nonce + CRC error
- If CRC error and 4 bytes received → detected = true

### Finding 4: NTAG203 Works Because...
The MF Ultralight detection uses simple page reads (no auth). See `mf_ultralight_poller_detect()`:
```c
MfUltralightError error = mf_ultralight_poller_read_page(instance, 0, &read_page_cmd_data);
protocol_detected = (error == MfUltralightErrorNone);
```
Simple read without authentication challenge-response = less sensitive to timing issues.

## Fix Strategies

### Fix 1: MIFARE Detection Improvements
**Option A (Recommended)**: Add retry logic with longer timeouts for MIFARE Classic detection
- In `mf_classic_poller_detect()`, if first attempt fails with communication error, retry after brief delay
- Consider adding more retries in PN532 I2C layer

**Option B**: Add delay between base protocol detection and child protocol detection
- In `nfc_scanner_state_handler_detect_children_protocols()`, add small delay before allocating child poller

**Option C**: Check if I2C needs reset between detection cycles
- Investigate if `furi_hal_i2c_bus_reset()` could be used safely here

### Fix 2: Crash on Exit
**Option A (Recommended)**: Ensure poller cleanup in scene exit handlers
- Add poller stop/free to `nfc_protocol_support_scene_read_on_exit()` as safety net
- Ensure all poller allocations have matching cleanup in exit paths

**Option B**: Add NULL checks in poller event handlers
- Verify `instance->poller` is non-NULL before calling stop/free

**Option C**: Check for race condition between poller callback and scene exit
- Ensure poller callback cannot fire after scene exit cleanup
