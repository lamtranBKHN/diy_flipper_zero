## "Crash" on NFC Read — Analysis

### Log Summary

The log shows normal NFC scanning behavior with **no card present**:

| Time | Event |
|------|-------|
| 24163 | PN532 init begins |
| 24846 | PN532 initialized OK over I2C |
| 24962 | HAL ready via PN532 backend |
| 39982 | 40 NFC card plugins loaded |
| 39989–41160 | Scanner polls 3 protocols (A, B, Felica) — all time out with "no targets found" |
| 41160 | Log ends after Felica poller freed |

### Key Observations

1. **PN532 initializes correctly** — single-read fix works
2. **No crash message** — no HardFault, no assertion failure, no watchdog reset in the log
3. **All protocol polls time out normally** — "FWT Timeout" and "no targets found" are expected when no card is near the antenna
4. **The "ViewPort lockup" warning at 36882ms** is pre-existing and expected on the DIY board (SPI1 sharing documented in AGENTS.md)

### Probable Cause

The scanner iterates through 3 protocols with a polling timeout of ~2-3 seconds each. During this time the UI displays "Scanning…" and appears frozen. The total scan cycle takes approximately 8–12 seconds. The user likely interpreted this delay as a crash because the screen appears non-responsive during the scan loop.

### How to Confirm

1. Hold an NFC tag (ISO14443-3A card, sticker, or phone with emulation) against the antenna during "NFC Read" — the scanner should detect it immediately
2. If no card is available, wait ~12 seconds — the device should return to the NFC start menu

### If the Device Actually Crashes

If the device hard-freezes or reboots (not just a UI delay), I need to see the serial output AFTER 41160ms. The current log ends at the point where the Felica poller is freed. A crash would produce additional messages (HardFault handler, watchdog, etc.) that would appear in the log after this point. Please share the full log from after the freeze/reboot occurs.

A post-41160 crash could be caused by:
- A null-pointer dereference in the scene transition after "no card found"
- An SPI bus contention between the display and PN532 (known issue on DIY board)
- A memory allocation failure during the 40 plugin loads

None of these are related to the P2P changes — they would be pre-existing issues in the NFC app or HAL.

### P2P Code Check

The P2P code (`snep_send`, `snep_receive`, `p2p_*`, `nfc_scene_p2p.c`) is NOT invoked during the NFC Read flow. These functions are only activated when the user explicitly selects "P2P Transfer" from the NFC menu. They cannot cause the NFC Read behavior.

### Hard Reboot Analysis

The log shows the scanner successfully cycles through all 3 protocols (A → B → Felica) before the reboot occurs. The reboot happens **after** the Felica poller finishes at 41160ms — during the scene transition back to the menu.

**Not caused by P2P changes.** All code paths exercised during NFC Read (PN532 init, scanner, poller, event system) are unchanged from the pre-P2P codebase. My changes only touched:
- `lib/nfc/p2p/*` — not invoked during scanning
- `applications/main/nfc/scenes/nfc_scene_p2p.c` — not entered
- `furi_hal_pn532.c` — single-read instead of two-phase (more reliable, not less)

**Most likely root cause**: The PN532 scanner activates the RF field for extended periods during the multi-protocol poll cycle (~12 seconds). On a DIY board, this current draw can cause a brown-out reset if the 3.3V regulator is marginal. This is supported by:
1. The timing (reboot happens after several protocol polls, not immediately)
2. `ViewPort lockup` warnings during the scan (SPI bus contention increases current draw)
3. The re-boot pattern (no HardFault messages, device just restarts)

**Verification**: If you have an NFC tag, hold it to the antenna immediately after selecting "NFC Read". If the tag is detected instantly (before the scan can time out), the reboot won't occur because the RF field is deactivated after detection. A crash prevented by shortening the scan confirms it's a power draw issue.

## PN532 Not Detected — Root Cause Analysis

### Symptoms

Log shows `pn532_read_response: bad body[0]=0x00 (expected 0xD5)` during `GetFirmwareVersion`:
```
33065 [I][FuriHalPN532] PN532 I2C addr detected: 0x48
33065 [I][FuriHalPN532] PN532 detected at 0x48, querying firmware...
33121 [I][FuriHalPN532] PN532 firmware ACK received
33172 [D][FuriHalPN532] PN532 ready after 1 retries
33174 [E][FuriHalPN532] pn532_read_response: bad body[0]=0x00 (expected 0xD5)
```

### Root Cause

**Not caused by P2P changes.** The I²C two-phase read optimization in `furi_hal_pn532.c:246-297` (`pn532_read_response`) performs two sequential I²C reads:

1. **Phase 1** (line 264): `furi_hal_i2c_rx(..., hdr, 6, 150)` — reads the 6-byte frame header `[I2C_READY][PREAMBLE][STARTCODE1][STARTCODE2][LEN][LCS]`
2. **Phase 2** (line 277): `furi_hal_i2c_rx(..., &rx[6], remaining, 150)` — reads the body + DCS + postamble

Each `furi_hal_i2c_rx` call uses `FuriHalI2cBeginStart / FuriHalI2cEndStop` (line 281-282). The STOP condition between the two reads causes the PN532 to **reset its internal I²C buffer pointer**. Phase 2 then reads from byte 0 of the PN532's output buffer instead of continuing from where Phase 1 left off, returning the I²C status byte (`0x01`) or stale data at position `rx[6]`.

When `rx[6]` = `0x00` instead of `0xD5` (PN532_PN532TOHOST), the checksum at `frame[5]` check fails, and `pn532_read_response` returns `false`.

### Why It Was Working Before

The two-phase read was introduced to fix the "270-byte read inefficiency" (NFC Bugs Fixed #6, 2026-05-11). This optimization is **intermittently unreliable** — it depends on:
- PN532 firmware version (some versions handle I²C STOP/RESTART differently)
- I²C bus timing (affected by pull-up resistor values on the DIY board)
- The `furi_delay_ms(1)` at line 253 between ready check and read

### Fix Options

| Option | Effort | Risk | Description |
|--------|--------|------|-------------|
| **A. Disable two-phase read** | 10 min | Low | Force full-buffer read by removing the two-phase optimization. Costs 270 bytes per I²C read but is always reliable. |
| **B. Fallback on mismatch** | 15 min | Low | After two-phase read, if `frame[5] != 0xD5`, retry with a single full-buffer read |
| **C. Use `I2cBeginRepeatedStart`** | 30 min | Medium | Change the second phase read to use `FuriHalI2cBeginRepeatedStart` instead of `FuriHalI2cBeginStart` to avoid the STOP condition between reads. Requires `furi_hal_i2c_rx_ext` with custom begin/end signals. |
| **D. Increase delay** | 5 min | Low | Increase `furi_delay_ms(1)` to `furi_delay_ms(5)` at line 253 to give more settling time. May not fully fix the issue. |

**Recommended: Option A** — simplest and most reliable. The 270-byte read overhead is negligible (one extra I²C transfer per NFC transaction).

### File to Modify

`targets/f7/furi_hal/furi_hal_pn532.c:246-297` — `pn532_read_response()`.

Replace the two-phase read logic with a single `furi_hal_i2c_rx` call of `sizeof(rx)` bytes.
