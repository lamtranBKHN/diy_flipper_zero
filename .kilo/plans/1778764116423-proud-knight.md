## Robustness Improvement Areas

### 1. Error Handling & Recovery
**Priority: High**

#### 1.1 I2C Communication Robustness
- **File**: `furi_hal_pn532.c`
- **Current**: Uses retry logic only; no bus reset (desync fatal, **no HW RST pin exists**). `pn532_write_frame` has 3 flat retries @ 50ms, `pn532_read_ack` has 3 flat retries @ 5ms.
- **Constraints**: I2C bus reset via 9 SCL pulses desyncs PN532 permanently. Improvements limited to retry strategy and input validation only.
- **Improvement**: 
  - Exponential backoff for write retries: 50ms â†’ 100ms â†’ 200ms
  - Reduce I2C traffic on healthy bus by skipping drain on success
  - Add early abort check integration to avoid blocking on unrecoverable failures
  - Add max retries as `#define` (currently `PN532_I2C_RETRIES`)

#### 1.2 Target State Management
- **File**: `furi_hal_nfc_pn532.c`
- **Current**: `target_valid` flag set on detection, cleared on comm error. No explicit timeout tracking.
- **Improvement**:
  - Add timestamp-based target freshness check: if last exchange > 5s, force re-poll
  - Clear target state on ALL exchange failures (currently only `FuriHalPn532ErrorComm`/`Timeout`)
  - Add explicit `target_tick` field to track last successful exchange

#### 1.3 Error Propagation
- **File**: `furi_hal_pn532.c`, `furi_hal_nfc_pn532.c`, `lib/nfc/nfc.c`
- **Current**: `FuriHalPn532Error` has 5 codes (None, Timeout, Comm, InvalidAck, InvalidFrame). `FuriHalNfcError` has 8 codes. `nfc_process_hal_error()` in `nfc.c:88` collapses `FuriHalNfcErrorCommunication`, `FuriHalNfcErrorCommunicationTimeout`, `FuriHalNfcErrorBufferOverflow` all into `NfcErrorInternal`.
- **Improvement**:
  - Add `FuriHalPn532ErrorBufferOverflow` for when `payload_len > rx_size`
  - Use `FuriHalNfcErrorCommunicationTimeout` in PN532 paths instead of `FuriHalNfcErrorCommunication` for actual timeouts
  - Keep `FuriHalNfcErrorCommunication` for I2C bus errors, `FuriHalNfcErrorBufferOverflow` for oversized responses
  - Preserve `FuriHalPn532Error` in diagnostic log for debugging

### 2. Timeout & Retry Logic
**Priority: High**

#### 2.1 Configurable Timeouts
- **File**: `furi_hal_pn532.c`
- **Current**: Fixed timeouts (150ms for ACK, 250ms for fw/SAM, 300ms for poll/auth, 500ms for ISO14443A, 1000ms for data exchange)
- **Improvement**:
  - Add `#define` constants for each timeout with documented rationale
  - Longer timeout (1500ms) for MIFARE Classic 4K (more data)
  - Shorter timeout (100ms) for presence detection re-poll
  - Timeout values as module-level constants (not `pn532_read_ack` literal 150)

#### 2.2 Retry Strategy
- **File**: `furi_hal_pn532.c`
- **Current**: 3 retries each in `pn532_write_frame`, `pn532_read_ack`, `pn532_read_response`. All flat delays.
- **Improvement**:
  - Progressive backoff: 50ms â†’ 100ms â†’ 200ms for writes
  - Progressive backoff: 5ms â†’ 10ms â†’ 20ms for reads
  - Distinguish retryable (I2C NACK/timing) vs non-retryable (invalid frame header, checksum failure) errors
  - Configurable max retries via `#define PN532_I2C_RETRIES 3` (already exists)

### 3. Protocol Detection Robustness
**Priority: Medium**

#### 3.1 Scanner Timing
- **File**: `nfc_scanner.c`
- **Current**: 300ms delay (`NFC_SCANNER_IDLE_DELAY_MS`) when all base protocols fail detection
- **Improvement**:
  - Reduce idle delay to 150ms for faster re-scan on tag approach
  - Dynamic timing: fallback to 300ms only after 3 consecutive empty scans
  - Add `furi_thread_flags_get()` abort check inside the idle delay loop

#### 3.2 Collision Handling
- **File**: `nfc_scanner.c`, `furi_hal_nfc_pn532.c`
- **Current**: Sequential protocol polling in fixed order. PN532 returns `NbTg` from InListPassiveTarget but only first target parsed.
- **Constraint**: PN532 response format for InListPassiveTarget is limited â€” only one target's full data is returned per poll. Multi-tag support requires multiple poll cycles.
- **Improvement**:
  - Priority ordering: ISO14443A first (most common), then FeliCa, then ISO14443B
  - Detect multiple tag count from `response[1]` (NbTg) and log warning
  - If NbTg > 1, re-poll to get next target after processing first

### 4. Logging & Debugging
**Priority: Medium**

#### 4.1 Debug Level Control
- **File**: Multiple NFC files
- **Current**: LOG_T, LOG_D, LOG_I, LOG_W, LOG_E levels used
- **Improvement**:
  - Runtime log level control via API
  - Module-specific logging (NFC, PN532, Protocol)
  - Timestamp correlation for multi-packet operations

#### 4.2 Diagnostic State
- **File**: `furi_hal_nfc_pn532.c`
- **Improvement**:
  - Track last error, last successful operation
  - Add diagnostic query API for debugging tools

### 5. Memory Safety
**Priority: Low**

#### 5.1 Buffer Bounds Checking
- **File**: `furi_hal_pn532.c`, `furi_hal_nfc_pn532.c`
- **Current**: Basic size checks in place. Stack allocation in `pn532_tx_custom_parity`: `uint8_t tx_bytes[PN532_MAX_FRAME_SIZE]` (192 bytes on stack).
- **Concern**: 192-byte stack buffer in ISR-invoked path â€” verify call chain for ISR safety.
- **Improvement**:
  - Add compile-time asserts for buffer sizes (`static_assert`) for critical arrays
  - Move `pn532_tx_custom_parity` stack buffer to static (mutex-protected) or heap if call chain ISR-unsafe
  - Validate input parameters more strictly with early-return on NULL
  - Canary values for critical buffers (debug builds, `FURI_DEBUG` only)

## Known Issues

### ~~Bug: Target freshness re-poll triggers immediately after detection~~ âś… FIXED (2026-05-14)

**Root Cause**: `furi_hal_nfc_pn532_trx_short_frame()` set `target_valid = found` but did NOT set `target_tick = furi_get_tick()`. When the next frame entered `exchange_internal`, it saw `target_tick == 0`, computed age as `furi_get_tick() - 0` which far exceeded the 5000ms timeout, triggering a spurious re-poll.

**Secondary Bug**: The `effective_tx_len < 3` guard blocked valid 2-byte SDD anticollision frames `[0x93, 0x20]` when re-poll was triggered, returning `FuriHalNfcErrorCommunication` instead of forwarding to InDataExchange.

**Fixes applied**:
1. `furi_hal_nfc_pn532.c:394` â€” Added `target_tick = furi_get_tick()` after `target_valid = found`
2. `furi_hal_nfc_pn532.c:440` â€” Guard now checks `!target_valid && effective_tx_len < 3`, only rejecting short commands when genuinely no target exists
