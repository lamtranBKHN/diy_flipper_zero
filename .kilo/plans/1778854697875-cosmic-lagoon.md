# Plan: NFC-10 — Peer-to-Peer (LLCP/SNEP) for PN532

## Goal

Enable NDEF exchange between the Flipper Zero and Android phones (or other NFC peers) using NFC Forum P2P over PN532's NFC-DEP (ISO 18092) mode. The flipper appears as an NFC peer and can send/receive NDEF messages (URLs, text) via LLCP + SNEP.

## Architecture

```
App (Scene)          ── sends NDEF URL/text, shows received NDEF
  │
  ▼
SNEP Layer           ── wraps NDEF in SNEP PUT/GET, checks responses
  │
  ▼
LLCP Layer           ── SYMM, CONNECT, CC, I, RR, DISC, DM PDU handling
  │
  ▼
MACLink (HAL)        ── tgInitAsTarget (DEP mode), tgGetData, tgSetData
  │
  ▼
PN532 (furi_hal_pn532) ── I²C frames, TGINITASTARGET/TGGETDATA/TGSETDATA
```

## Differences from existing Type 4 Tag Listener

| Aspect | Type 4 Tag (current) | P2P (new) |
|--------|---------------------|-----------|
| PN532 mode | `TgInitAsTarget` mode=0x05 (ISO14443A target) | `TgInitAsTarget` mode=0x00 with LLCP magic (DEP) |
| Protocol | ISO14443-4A → Type 4 Tag | NFC-DEP (ISO 18092) → LLCP → SNEP |
| Data exchange | Reader-initiated (APDU-based) | Peer-to-peer (symmetric) |
| Initiation | Reader polls, tag responds | Either side can initiate |
| NDEF direction | Reader writes, tag reads | Both directions (put/get) |
| Connection | State-less APDU exchange | Connection-oriented (CONNECT/CC/DISC) |

## Risk Assessment

- **PN532 firmware version**: P2P requires PN532 firmware v1.6+. Check with `GetFirmwareVersion`. If older, P2P is unreliable.
- **Timing**: DEP symmetry interval is 50ms. The PN532 handles this internally through its hardware state machine, so the MCU doesn't need precise timing.
- **Buffer sizes**: NDEF messages are limited to ~240 bytes by the PN532's DEP payload limit (255 - 2 LLCP header - 6 SNEP header = 247 bytes max). This is sufficient for URLs and short texts.
- **Phone compatibility**: Some Android phones may not support P2P correctly. Test with Google Pixel, Samsung Galaxy series.
- **I²C polling**: The 200ms polling slice in the listener may cause latency issues. The DEP symmetry interval requires responses within ~50ms. May need to reduce the slice time or use IRQ-driven I²C for P2P mode.
