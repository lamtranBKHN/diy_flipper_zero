# Fix Device App Loading Crashes (Memory Exhaustion)

## Problem
Device crashes loading ANY app:
- `Not enough memory to load section data`
- `Error loading section '.debug_info'`
- `Not enough memory`

STM32WB55 RAM exhausted by ELF debug sections during app load.

## Comparison with Reference Project (diy_flipper_UBYTE_ssd1306_pcf8574)

### File Size Differences (FAPs)
| App | Current (bytes) | Reference (bytes) | Larger by |
|-----|-----------------|-------------------|-----------|
| nfc.fap | 312,808 | 273,488 | 14.4% |
| upython.fap | 450,324 | 259,872 | 73.3% ⚠️ |
| metroflip.fap | 650,416 | 510,120 | 27.5% ⚠️ |
| video_game_module_tool.fap | 642,340 | 628,888 | 2.1% |

### Key Findings
1. **Both projects have identical build code** (`fbt_extapps.py`, `extapps.scons`) - both already use `--strip-debug --strip-unneeded`
2. **Micropython library files differ** - reference has smaller source files (e.g., `.gitsubtree` 80 vs 81 bytes, `mp_flipper_modflipperzero.c` 32331 vs 33146 bytes)
3. **External apps are NOT submodules** - they're vendored as subtrees (gitsubtree)

### Root Cause Analysis
1. **FAP files are larger** in current project due to:
   - Different source code in micropython library (upython app)
   - Different upstream versions of external apps
2. **The `--strip-debug` in objcopy already works** - FAPs don't contain `.debug_*` sections
3. **The issue is code bloat**, not debug symbols

## Fix Approach

### Option A: Update external apps to reference versions
- Replace `applications/external/mp_flipper` with reference version
- Replace other oversized apps (metroflip, nfc, etc.)

### Option B: Investigate why apps fail to load despite size
- FAP size alone shouldn't cause "Not enough memory" if sections are loadable
- Check if memory fragmentation or other kernel changes affect heap

## Proper Fix Plan

### Root Cause
The current project and reference project use **completely different versions** of `applications/external`:
- **Current project**: Submodule commit `6d03fe32f` (newer, larger FAPs)
- **Reference project**: Vendored tree (no submodule) at commit `1b7ce492e` (older, smaller FAPs)

### Solution Options

#### Option A: Update Submodule (Recommended)
Update the submodule to the reference commit to match the working reference build:
```bash
# In the application external directory
git fetch origin
git checkout 1b7ce492e  # or appropriate parent commit
# Then rebuild FAPs
fbt fap_dist
```

#### Option B: Vendor External Apps (Like Reference)
Remove the submodule and vendor the external apps directly:
```bash
# Remove submodule
git submodule deinit -f applications/external
rm -rf .git/modules/applications/external
rm -rf applications/external
git rm -f applications/external
# Copy vendored version from reference
robocopy /path/to/reference/applications/external applications/external /MIR
```

#### Option C: Find Minimal FAP Commits
Search the Momentum-Apps history for commits that produce smaller FAPs:
```bash
# In applications/external submodule
git log --oneline --all --grep="upython\|micropython" --since="2025-01-01"
# Test builds at different commits to find minimal size
```
