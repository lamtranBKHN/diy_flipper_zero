# Plan: Fix NFC Read Freeze & MIFARE Classic Auth on PN532

## Goal
Fix MIFARE Classic 1K card reading and prevent scanner/UI freezes when card is removed mid-scan.

## Issues

### Issue A: MIFARE Classic auth always fails with `in_data_exchange status error: 0x14`
- **Root cause**: `exchange_internal` in `furi_hal_nfc_pn532.c` intercepts MIFARE auth (0x60/0x61) and builds a 12-byte command `[0x60, block, key(6), uid(4)]` that is sent via `InDataExchange`. The PN532's internal MIFARE auth handling returns 0x14 for ALL keys ‚Äî including factory default `ffffffffffff` ‚Äî making the dictionary attack (2311 keys) completely useless.
- **Log signature**: `in_data_exchange status error: 0x14` followed by `Target communication failed (err=2), marking invalid` repeated for every auth attempt.

### Issue B: Scanner enters infinite A/B/Felica loop after card removal
- **Root cause**: `nfc_scanner.c` has no maximum-round limit. The scanner cycles through ISO14443-A ‚Üí B ‚Üí FeliCa indefinitely when no card is present. The "Don't move" screen persists because the scanner never signals "no card detected" ‚Äî it keeps polling forever.
- **Log signature**: Repeated cycles of `poll_iso14443a: no targets found`, `pn532_exchange: response timeout (hardware)`, with 300ms delays between rounds, continuing until user presses Back.

### Issue C: ISO14443-4A ATS fails with `in_data_exchange status error: 0x0B` (deferred)
- **Root cause**: During child protocol detection, the ISO14443-4A poller tries to get ATS via InDataExchange after synthesized SDD/SELECT frames. The PN532's target state becomes inconsistent.
- **Log signature**: `in_data_exchange status error: 0x0B` during `Iso14443_4aPoller` ATS request.
- **Deferred**: Not critical ‚Äî child detection simply fails and falls back to the base protocol.

## Fixes

### Fix A1: Route MIFARE Classic auth through InCommunicateThru instead of InDataExchange

**File**: `targets/f7/furi_hal/furi_hal_nfc_pn532.c`

**What**: Replace `furi_hal_pn532_in_data_exchange` call in the `exchange_internal` 0x60/0x61 path with `furi_hal_pn532_in_communicate_thru`. Send only the raw 2-byte auth frame `[0x60/0x61, block_num]` ‚Äî without key or UID. The PN532 sends this raw frame to the card, the card responds with NT (4 bytes), and the NT is returned to the MIFARE poller without CRC append.

**Why**: The PN532's InDataExchange interprets 0x60/0x61 as a MIFARE auth command requiring 12-byte `[cmd, block, key(6), uid(4)]` format. This internal auth handling is buggy/fails. InCommunicateThru bypasses this by sending raw frames ‚Äî the card returns NT regardless of key correctness, and the MIFARE poller's host-side Crypto1 handles the actual authentication.

**Changes**:
1. Replace the auth_cmd[12] construction + `furi_hal_pn532_in_data_exchange` with a raw 2-byte frame sent via `furi_hal_pn532_in_communicate_thru`
2. On success, call `furi_hal_nfc_pn532_prepare_rx(rx_payload, rx_len, false, false)` ‚Äî **no CRC append** (NT isn't a standard ISO14443-3A frame)
3. Return immediately with `furi_hal_nfc_pn532_finalize_exchange(FuriHalPn532ErrorNone, true)`
4. On error, fall through to the existing error handling at line 602
5. Remove the unused `mf_auth_key`, `mf_auth_key_type`, `mf_auth_key_valid` fields from `FuriHalNfcPn532State` (or leave them for `furi_hal_nfc_pn532_mf_auth` which is called from `mf_classic_poller_detect`)

**Before (lines 559-587)**:
```c
// MIFARE Classic auth step 1 ‚Äî use full InDataExchange with key+UID
if(send_len >= 2U && (tx_bytes[0] == 0x60U || tx_bytes[0] == 0x61U)) {
    if(furi_hal_nfc_pn532.mf_auth_key_valid && furi_hal_nfc_pn532.target_valid) {
        uint8_t auth_cmd[12] = {0};
        auth_cmd[0] = tx_bytes[0];
        auth_cmd[1] = tx_bytes[1];
        memcpy(&auth_cmd[2], furi_hal_nfc_pn532.mf_auth_key, 6);
        size_t uid_copy = furi_hal_nfc_pn532.target.uid_len;
        if(uid_copy > 4) uid_copy = 4;
        memcpy(&auth_cmd[8], furi_hal_nfc_pn532.target.uid, uid_copy);
        if(uid_copy < 4) {
            memset(&auth_cmd[8 + uid_copy], 0, 4 - uid_copy);
        }
        err = furi_hal_pn532_in_data_exchange(
            furi_hal_nfc_pn532.target.target_number,
            auth_cmd, sizeof(auth_cmd),
            rx_payload, sizeof(rx_payload), &rx_len);
    } else {
        FURI_LOG_W(TAG, "MIFARE auth: no stored key or target invalid");
        err = FuriHalPn532ErrorComm;
    }
}
```

**After**:
```c
// MIFARE Classic auth ‚Äî use InCommunicateThru to bypass PN532 internal auth
// Send raw [0x60/0x61, block_num] frame; card returns NT (4 bytes)
if(send_len >= 2U && (tx_bytes[0] == 0x60U || tx_bytes[0] == 0x61U)) {
    uint8_t raw_auth[2] = {tx_bytes[0], tx_bytes[1]};
    err = furi_hal_pn532_in_communicate_thru(
        raw_auth, sizeof(raw_auth),
        rx_payload, sizeof(rx_payload), &rx_len);
    if(err == FuriHalPn532ErrorNone) {
        furi_hal_nfc_pn532_prepare_rx(rx_payload, rx_len, false, false);
        return furi_hal_nfc_pn532_finalize_exchange(FuriHalPn532ErrorNone, true);
    }
}
```

### Fix B1: Add max-empty-round limit to scanner

**File**: `lib/nfc/nfc_scanner.c`

**What**: Add `NFC_SCANNER_MAX_EMPTY_ROUNDS` (20 rounds ‚âà 18s). When `consecutive_empty_scans` reaches this limit, transition to `NfcScannerStateComplete` to fire the callback and stop the worker.

**Why**: Without a limit the scanner polls A/B/Felica forever. The UI shows "Don't move" with no progress and no way to exit except pressing Back.

**Changes in nfc_scanner.c**:
1. Add `#define NFC_SCANNER_MAX_EMPTY_ROUNDS 20`
2. In `nfc_scanner_state_handler_try_base_pollers`, after `consecutive_empty_scans++` (line ~153) add:
```c
if(instance->consecutive_empty_scans >= NFC_SCANNER_MAX_EMPTY_ROUNDS) {
    instance->state = NfcScannerStateComplete;
    break;
}
```
3. No changes to `nfc_scanner.h` ‚Äî the `Detected` event is fired with `protocol_num = 0` (the app handles zero protocols gracefully).

## Risk Assessment

- **Fix A1** (InCommunicateThru for auth): Low risk. InCommunicateThru is a standard PN532 command that sends raw frames. The response (NT) is returned as raw bytes. The MIFARE poller's host-side Crypto1 handles the rest. If InCommunicateThru also fails, auth reverts to the existing error handling ‚Äî no regression.
- **Fix B1** (scanner timeout): Low risk. The timeout stops the scanner after ~20 empty rounds (~18s). A `Detected` event fires with `protocol_num=0`. The NFC app must handle zero protocols gracefully (show "no card found" or return to menu).

## Testing

1. **Fix A1**: Place MIFARE Classic 1K card (ATQA=0004 SAK=08). Verify:
   - Scan detects card
   - Auth succeeds with factory default key `ffffffffffff`
   - Dictionary attack finds keys
   - Card data is read successfully
   
2. **Fix B1**: Start NFC Read without a card. Verify:
   - Scanner cycles through A/B/Felica for ~20 rounds
   - Scanner stops with timeout event (or stops silently)
   - No indefinite freeze
   - UI returns to menu or shows "No card found"

3. **Regression**: Verify other NFC protocols still work:
    - ISO14443-3A (Type A cards, e.g. NTAG)
    - ISO14443-3B (if available)
    - FeliCa (if available)
    - Child protocols: ISO14443-4A, NTAG, MIFARE Ultralight
4. **Fix B1 edge case**: Start NFC Read, don't place a card. Verify scanner stops after ~18s and app returns to menu (no indefinite freeze).
